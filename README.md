# Полиморфные абстрактные типы данных (Лабораторная работа №3 - MAtrixies)

Система линейных структур данных на C++, предоставляющая унифицированный интерфейс `Sequence<T>` для работы с динамическими массивами и связными списками, с поддержкой парадигм Mutable и Immutable объектов

## Сборка и запуск

Проект использует систему сборки CMake. Все команды выполняются из корневой директории проекта

**1. Конфигурация и сборка проекта**
```bash
cmake -B build             # Конфигурация (выполняется единожды или при изменении CMakeLists.txt)
cmake --build build        # Компиляция проекта
```
2. Запустить только исполняемую программу (интерфейс):
```
./build/src/Lab3_app       # Запуск основного исполняемого файла
```

3. Запуск и фильтрация тестов
```bash
cd build && ctest --output-on-failure                         # Запуск всех тестов с выводом ошибок
./build/tests/run_tests --gtest_filter="SquareMatrixTest*"    # Запуск конкретного набора тестов (Google Test)
```

## Архитектура и детали реализации

### Интеграция с предыдущими работами (Lab 2)
Для хранения набора сгенерированных матриц в пользовательском интерфейсе переиспользуется абстракция из второй лабораторной работы  `MutableArraySequence<IMatrix<T>*>`. Это доказывает универсальность написанных ранее контейнеров и соблюдение принципов переиспользования кода (DRY). В качестве хранилища в классе матрицы используется слой памяти `dynamic_array`, о чем позже.

### Принцип подстановки Лисков (LSP)
`Дочерний класс должен дополнять, а не замещать поведение родительского класса.`
Любая матрица (квадратная, диагональная, треугольная) может быть передана по указателю `IMatrix<T>*`. Полиморфизм гарантирует, что вызывая метод `get(i, j)`, мы получим корректный элемент, а вызывая `set(i, j, val)` на невалидной позиции (например, вне главной диагонали у `DiagonalMatrix`), мы получим строго специфицированное исключение `std::logic_error`, защищающее математическую целостность объекта.

### Тип Complex<T>
Класс комплексных чисел Complex<T> был глубоко интегрирован в проект:

1. Парсинг: Написан "умный" перегруженный `operator>>`, который позволяет считывать числа из консоли в гибких форматах: `(re, im)`, `(re)` или просто `re` (для случаев, когда мнимая часть равна нулю).

2. Абстракция модуля: Для корректной работы алгоритма Гаусса с выбором ведущего элемента (где требуется сравнение по модулю), добавлено пространство имен `custom_math::abs()`, унифицирующее получение абсолютного значения как для `double`, так и для `Complex`.

### IMatrix<T>
#### Схема наследования
```bash
IMatrix<T>               ← Абстрактный базовый класс (интерфейс)
   ├── Matrix<T>         ← Прямоугольная плотная матрица
   │   ├── SquareMatrix<T> ← Квадратная матрица (rows == cols)
   ├── TriangularMatrix<T>  ← Оптимизированное хранение n*(n+1)/2 элементов
   └── DiagonalMatrix<T>    ← Хранение только главной диагонали (n элементов)
```

### Защита целостности памяти (Отсутствие operator[][])

В архитектуре принципиально не реализован доступ по двойным квадратным скобкам `matrix[i][j]` для записи. Возврат ссылки на внутреннюю память (которая требуется для `operator=`) разрушил бы инкапсуляцию таких классов, как `DiagonalMatrix` и `TriangularMatrix`. Если пользователь попытается записать значение вне разрешенной зоны в обход проверки метода `set()`, структура сломается. Для удобства чтения добавлен `operator()(i, j) const`, являющийся синтаксическим сахаром над `get(i, j)`.

### Инициализация нулем (Value Initialization)
Вместо использования статических констант или `AlgebraTraits`, везде в коде используется универсальная инициализация `T{}`. Она гарантирует, что для примитивов (`double`) будет создан `0.0`, а для пользовательских классов (`Complex`) будет вызван конструктор по умолчанию, создающий аддитивный ноль.

### SystemOfEquations<T>
Ядро для решения систем линейных алгебраических уравнений (СЛАУ). Все алгоритмы возвращают `Vector<T>` по значению (Правило Нуля), исключая утечки памяти.

#### Сеттеры и управление кэшем
Реализовано интеллектуальное кэширование PLU-разложения.
- `set_matrix(A)` / `set(A, b)` — сбрасывают флаг `lu_ready = false`, так как матрица коэффициентов изменилась.
- `set_rhs(b)` — не сбрасывает кэш. Это позволяет менять только вектор свободных членов и решать новые системы за $O(N²)$ вместо $O(N³)$.

#### Реализованные алгоритмы решения:
1. `solve_gauss()` (Наивный Гаусс)

Классический алгоритм без перестановок. Добавлена строгая проверка не только в цикле, но и для последнего диагонального элемента $A_{n−1,n−1}$​ на равенство нулю, чтобы избежать скрытого деления на ноль и возникновения `NaN` или `Inf`.

2. `solve_gauss_with_pivot()`

Решение СЛАУ методом Гаусса с частичным выбором ведущего элемента. Перед исключением переменной алгоритм ищет максимальный по модулю элемент в столбце и меняет строки местами, минимизируя ошибку потери значащих цифр (вычислительную неустойчивость). Написан независимо от PLU для чистоты бенчмарков.

3. `solve_lu()` (Чистое LU-разложение)

Реализовано строго по ТЗ. Алгоритм Дулиттла факторизует матрицу $A=L⋅U$ без использования перестановок. Служит демонстрацией того, как классический алгоритм из лекций "ломается" при наличии нуля на диагонали.

4. `solve_plu()` (Продвинутое PLU-разложение)

Оптимизированный "движок" решения $PA=LU$.

- Использует ключевое слово `mutable` для скрытого кэширования матриц `P`, `L` и `U` внутри const метода.

- Матрица `U` хранится как плотная `SquareMatrix`, чтобы безопасно выполнять операции `swap_rows()`.

- Матрица `L` хранится как `TriangularMatrix` для экономии памяти.

- Матрица `P` оптимизирована до одномерного массива индексов (`DynamicArray`).

#### Фабрики

- `random(n, seed)` — генерация систем со случайными коэффициентами для бенчмарков.
- `hilbert(n)` — генерация СЛАУ на базе плохо обусловленной матрицы Гильберта $H_{ij} = \frac{1}{i+j-1}$ с точным вектором решения $x^* = (1, 1, \dots, 1)^T$. Позволяет измерить относительную погрешность $\frac{\|x_{approx} - x_{exact}\|}{\|x_{exact}\|}$​.
